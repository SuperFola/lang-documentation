Abstract

This KEP describes the purpose of KEPs, and defines guidelines for writing one.

Content

Definition

KEP stands for Kafe Enhancement Proposal. The sole purpose of a KEP is to define a (new) feature for Kafe, in a concise and technical way. A feature can be either a new mechanism for the language or a description of a bug (and why it should be considered as one) and how it could be fixed.

All KEPs can be found in https://github.com/Kafe-lang/enhancement-proposals on GitHub, alongside their revision history.

KEP types

As said before, a KEP can be either : 1. a new mechanism for the language 2. a bug fix

From Markdown to HTML

This section is only useful if you want to check if any errors where triggered, which mean either the structure or the format wasn't respected.

Download the Python script md2html.py from the GitHub and run :

user:~$ python md2html.py kep-draft.md

Note: Python >= 3.6 needed

It will produce a kep-draft.html which you will be able to visualize in your browser by running : 

user:~$ sensible-browser kep-draft.html

KEP workflow

The Kafe BDFL

The Kafe BDFL - where BDFL stands for Benevolent Dictator For Life - is Alexandre Plateau, the original creator of the language.

Begin with an idea to improve the language

The process of writing a KEP begins with an idea for Kafe. A single KEP must contain a single idea (or bug fix) to improve the language.

If a KEP appears to be too broad, it should be rejected and split in multiple ones.

To save you time, before writing your KEP, talk about it on our Discord server to know if it could really be useful (or, in case of a bug fix, if it really appears as one to the rest of the community). This way we will be able to help you writing your KEP, which will make the reviewing process shorter for both of us.

Submit your KEP for review

Once your KEP is written and once you think it's ready to be shown, fork the KEP repository, and add your KEP named as kep-draft.md.

Then you can create a new pull request on GitHub to submit it to reviewers, who will review the structure, format, and errors.

Once approved, it will be given a number and merged into the repository.
